Explainer · all_in_one_grpo.py

Teaching a small model to reason
without burning its core

LF-GRPO (Layer-Frozen GRPO) is the experiment behind Surtur. It trains a 3B model to solve math on a single T4 GPU — but only lets gradients touch the last few layers. The core stays frozen and insulated. This page walks through exactly how the script does it.

§1 · The big idea

Confine the fire, freeze the core

Normal fine-tuning backpropagates through every layer. That slowly erodes the model's central reasoning ability — the research called it "Central Engine Disruption." LF-GRPO instead restricts training to the behavioral periphery (the final layers) and proves, mid-training, that the frozen core received zero gradient.

The result that justified Surtur: layer-frozen training reached 50% math accuracy vs 42% for full-layer training — and it killed reward-hacking and halved latency. Same model, same data, same GPU. The only difference was where gradients were allowed to flow.

§3 · Inside one GRPO step

Groups, rewards, relative advantage

GRPO needs no critic model. For each prompt it generates several answers, scores them, and compares each answer to the group average — pushing up the ones that beat the average.

1 · Prompt
Take one math question.
2 · Generate
Sample num_generations = 4 completions (the "group").
3 · Reward
Each completion scored by the reward functions (§4).
4 · Advantage
Compare to the group: A = (r − mean) / std. Better-than-average answers are reinforced.
5 · Update
One policy-gradient step on the trainable layers only — then repeat.
§4 · The reward functions

How "good" is measured

Every function shares the same shape: (prompts, completions, target_answer) → list[float]. They are combined and weighted (§5).

format

format_reward_fn

1.0 for one clean <think> block; 0.3 multi-block; 0.2 unclosed; 0.0 none.

correctness

math_correctness_reward_fn

Extracts the final number (after </think> or \boxed{}) and checks it equals the target.

p-grpo

p_grpo_format_reward_fn

Same as format, but zeroed if the answer is wrong — stops the model from "looking smart" while being wrong.

step-grpo

step_grpo_reward_fn

If correct, reward decays by 0.99^steps (steps = "wait/hmm/actually" tokens) and penalizes extra blocks. Rewards concise reasoning.

§5 · Two-stage training

Learn format first, then correctness

The combined reward blends format and correctness, but the weights flip after stage_steps = 50.

Stage 1 · steps 0–50

format weight 1.0correct 0.1

Primer: teach the model to wrap reasoning in <think>…</think>.

Stage 2 · steps 51+

format weight 0.2correct 1.0

Now optimize for actually getting the math right.

§6 · The frozen-core guardrail

Proof the core stayed asleep

This is the heart of Surtur's "preservation" promise. A callback fires after the first backward pass and inspects every parameter.

StepTrackerCallback · on_substep_end (step 0)

For each parameter: is it frozen (requires_grad = False) or active? Sum the gradient norm of the frozen ones. If any frozen parameter has gradient > 1e-9, the script raises:

RuntimeError("Gradient insulation check failed: Frozen layers received gradients.")

If all frozen gradients are exactly 0.0 → "100% gradient insulation confirmed." The core is proven untouched. This is the exact guarantee your config.py is designed to feed.

§7 · Resolving the layers

From last_4 to concrete indices

The script turns a preset into a layer list — and this is exactly the logic your config.py implements, generalised to any model depth.

layer 27 · trainable (fire)
layer 26 · trainable (fire)
layer 25 · trainable (fire)
layer 24 · trainable (fire)
layer 23 · frozen (core)
layer 22 · frozen (core)
…
layer 0 · frozen (core)

For a 28-layer model, last_4 resolves to:

list(range(num_layers − 4, num_layers))
→ [24, 25, 26, 27]

LoRA is then attached only to those layers; every other layer's requires_grad stays False. The same code accepts "24,25,26,27" or any custom set — which is what your Surtur config module adds on top.
