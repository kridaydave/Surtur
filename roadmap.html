EAL · Surtur

Surtur — the fire kept at the edge

Epoch's spatial training & capability-preservation runner. Post-training that confines the fire of parameter optimization to the behavioral periphery (the final ~10–15% of depth — e.g. layers 24–27 on a 28-layer model), keeping the model's core logic engine frozen and insulated.

Hold core-eval within ≤2% of the frozen baseline while delivering a measurable alignment gain — on one GPU at ≤30% the compute.

Exact layer numbers shift with model size — the principle is relative depth, not fixed indices.

§1 · Vision & north-star

Make capability-preserving post-training a cheap, reproducible habit

Align models on a single GPU without silently burning the core they already earned. Success is defined by outcomes that move a number — not by features shipped.

§2 · The problem & the blocker

One blocker gates the entire roadmap

Post-training backpropagates across the entire model, degrading the central logic engine. Epoch researchers want alignment without re-earning capabilities they already paid for — and without a GPU cluster.

Action · research leads, this week

Specify (1) the core-capability eval suite — the "do not regress this" set — and (2) the retention / alignment thresholds that define pass vs. fail. Without these two definitions, Phase 0 cannot pass or fail, and every downstream phase is built on an unverifiable premise.

§3 · Strategic principles

The lines we don't cross

A roadmap is what we chose not to do. These five hold for every phase.

1

Preservation is the product

If a run improves alignment but quietly degrades the core, it has failed. The core-eval suite is sacred.

2

Single GPU is the constraint

Anything that requires a cluster is out of scope for v1.

3

Reproducibility is non-negotiable

A number we can't rerun isn't evidence.

4

Constraint strategy, not new architecture

Surtur is a where-gradients-flow decision. We do not invent optimizers or attention variants.

5

Say no to good ideas

The cut lines below are deliberate. A roadmap is what we chose not to do.

P0

Validate the core claim

"Does confining gradients to layers 24–27 preserve the core AND align?" — currently unverified.

The MVP — one head-to-head experiment

Arm A — Surtur: layers 24–27 trainable, core frozen (no grad). Arm B — Full fine-tune: standard SFT/GRPO across all layers. Arm C — Frozen baseline: no training. Arm D — Untrained-but-aligned reference (if available). All arms use the same data, same seed, same objective.

Must

  • Fixed target-layer config (24–27) + frozen-core gradient masking
  • Core-capability eval suite (defined by research leads)
  • Alignment eval suite (defined by research leads)
  • Single shared training pipeline for all arms

Should

  • Per-layer gradient-norm logging
  • Ablation: test layers {22–25}, {24–27}, {26–29}

Won't

  • UI, multi-GPU, novel methods
  • Hyperparameter sweeps

Definition of Done

  • Each arm reproducible from a single command (same data + seed).
  • Core-eval: Surtur (A) ≥ Frozen baseline (C) − 2%.
  • Alignment-eval: Surtur (A) > Untrained reference (D).
  • Results table committed with config + metric provenance.

Kill / revisit if

  • Core degrades beyond threshold AND only full fine-tune (B) reaches the target.
  • The spatial-constraint premise fails → pivot or stop. Do not proceed to P1.

Estimated effort: 1 researcher + 1 engineer, ~1–2 weeks including eval-suite definition.

P1

Core runner (the engine)

"Can we make the validated experiment a reusable, config-driven runner?" — only starts if P0 passes.

The MVP

A config-driven training runner that accepts a target-layer spec, freezes the core, and runs SFT or GRPO on a single GPU, producing checkpoints and logs.

Must

  • Config-driven target-layer selection
  • Frozen-core masking (no-grad on core)
  • Single-GPU optimization (grad accumulation / CPU offload)
  • SFT and GRPO training loops
  • Checkpoint + clean resume

Should

  • Live gradient-norm + capability-trend logging
  • Guardrail: hard error if a frozen layer receives grad
  • Config schema validation

Won't

  • Multi-GPU / distributed training
  • Novel optimizers / custom attention
  • Auto layer-band selection

Definition of Done

  • A probe confirms only layers 24–27 have non-zero gradients.
  • Full SFT and full GRPO each complete on one GPU within budget.
  • Resume from 60% interruption restores state to completion.
  • Retention matches the P0 validated result (regression guard).

Kill / revisit if

  • Runner can't hit the compute budget, or retention regresses vs. P0.

Estimated effort: 1–2 engineers, ~3–4 weeks.

P2

Reproducibility (Anchor)

"Can anyone rerun any result, anywhere?" — runs in parallel with late P1.

The MVP

A pinned, versioned, one-command reproduction path for every published result.

Must

  • Pinned environment (lockfile or container image)
  • Seed control + dataset versioning (hash-pinned)
  • Run command + config hash captured per result
  • Results registry (run → config → metrics)

Should

  • Deterministic dataloader / sampler ordering
  • One-command "reproduce result #ID" entrypoint

Won't

  • CI that auto-runs all configs
  • Public result portal

Definition of Done

  • A fresh environment reproduces metrics within tolerance from one command.
  • Every metric links to exact config + data version + seed.
  • A new researcher reproduces a P0 result without author help.

Kill / revisit if

  • Reproducibility consistently requires author intervention.

Estimated effort: 1 engineer, ~2 weeks (overlaps P1).

P3

Internal dashboard (the UI)

"Can researchers use it day-to-day without CLI heroics?" — design starts when P1 interface stabilizes; ship only after P2.

The MVP

A web dashboard where a researcher submits a run config, watches live training, and sees the capability-preservation delta versus the frozen baseline — without leaving the page.

Must

  • Submit a run config (target layers, method, dataset)
  • Live training-metrics view (loss, grad norm, step)
  • Capability-preservation comparison vs. frozen baseline
  • Run history + status (queued / running / done / failed)
  • Authentication via existing Epoch SSO

Should

  • Per-layer gradient visualization
  • Side-by-side comparison of two runs
  • Artifact links to the P2 results registry

Won't

  • Multi-tenant auth / external sharing
  • Run scheduling / queueing
  • Billing / quotas / chargeback

Definition of Done

  • SSO-authenticated researcher launches a run from the UI.
  • Live metrics render within ≤10s staleness.
  • Completed run shows core-retention delta without leaving the page.
  • Dashboard links a run to its P2 registry entry.

Kill / revisit if

  • Researchers keep using the CLI despite the dashboard.

Estimated effort: 1–2 engineers (frontend + backend), ~4–6 weeks including Atlas design.

P4

Adoption & hardening

"Is Surtur the default, not a curiosity?" — starts after P3 ships.

Must

  • Template configs for common Epoch alignment jobs
  • One end-to-end worked example (doc)
  • Track adoption + median retention

Should

  • Auto layer-band suggestion from a quick probe run
  • Integration with Epoch's existing eval/reporting

Won't

  • External community features
  • Custom model architectures

Definition of Done

  • New researcher completes a run template→result in < 1 day.
  • Adoption target met for two consecutive months.
  • Median retention ≥ the P0 threshold.

Kill / revisit if

  • Adoption stays < 20% after active onboarding → re-interview users.

Success: ≥60% of Epoch alignment runs on Surtur within 2 quarters; median retention ≥ P0 threshold; GPU-cost ratio ≤30%.

§11 · Explicitly NOT building

The cut line

Out of scope

  • A hosted / productized platform with external customers
  • An open-source / community release
  • Multi-GPU / distributed training
  • Multi-tenant auth, external sharing, billing/quotas
  • Run scheduling / queueing
  • Novel ML methods / optimizers / architectures

In scope (the bet)

  • Internal research tool, single GPU
  • A constraint strategy: where gradients flow
  • Reproducible, registry-backed numbers
  • A dashboard that makes retention visible
